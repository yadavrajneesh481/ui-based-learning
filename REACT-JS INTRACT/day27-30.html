2Ô∏è‚É£ Core Concepts: createSlice, configureStore

üç∞ createSlice

Definition: A function that accepts an initial state, an object of reducer functions, and a "slice name", and automatically generates action creators and action types.

Kya karta hai? Ek hi jagah pe actions aur reducers define karo. RTK automatically action creators aur types bana deta hai!

‚öôÔ∏è configureStore

Definition: A friendly abstraction over the standard Redux createStore that adds good defaults (Redux DevTools, thunk middleware) automatically.

Kya karta hai? Store setup ko super easy bana deta hai. DevTools aur middleware automatically configure ho jate hain!

üîÑ createReducer

Definition: A utility that lets you define reducers as lookup tables of action types to case reducer functions, with built-in Immer support.

Kab use karein? Jab aapko createSlice se zyada control chahiye. Usually createSlice hi kaafi hai!

üé¨ createAction

Definition: A helper function for defining Redux action creators with a given action type string.

Kab use karein? Rarely! createSlice automatically actions bana deta hai. Manual control ke liye use karo.

3Ô∏è‚É£ Async Thunks - API Calls Handle Karna

createAsyncThunk: A function that accepts a Redux action type string and a callback function that should return a promise. It generates promise lifecycle action types (pending, fulfilled, rejected) and dispatches them based on the promise status.

Problem: Redux mein async operations (API calls) directly nahi kar sakte. Reducers pure functions hone chahiye.

Solution: createAsyncThunk! Ye automatically teen actions generate karta hai - pending (loading), fulfilled (success), rejected (error). Aapko sirf async function likhna hai! üéØ

// createAsyncThunk Example import { createSlice, createAsyncThunk } from '@reduxjs/toolkit'; // 1. Create Async Thunk export const fetchUsers = createAsyncThunk( 'users/fetchUsers', // Action type prefix async (userId, thunkAPI) => { try { const response = await fetch(`/api/users/${userId}`); return await response.json(); // Fulfilled payload } catch (error) { return thunkAPI.rejectWithValue(error.message); } } ); // 2. Handle in Slice const usersSlice = createSlice({ name: 'users', initialState: { data: [], loading: false, error: null }, reducers: {}, extraReducers: (builder) => { builder // Pending - API call start hui .addCase(fetchUsers.pending, (state) => { state.loading = true; state.error = null; }) // Fulfilled - Success! .addCase(fetchUsers.fulfilled, (state, action) => { state.loading = false; state.data = action.payload; }) // Rejected - Error! .addCase(fetchUsers.rejected, (state, action) => { state.loading = false; state.error = action.payload; }); } });

5Ô∏è‚É£ Normalizing State - Data Ko Efficiently Store Karna

State Normalization: A pattern of organizing data in the Redux store by storing items in a flat structure with IDs as keys, rather than nested arrays. This improves performance and makes updates easier.

Problem: Nested data structures mein updates slow aur complex ho jate hain. Agar aapko ek user update karna hai jo deeply nested hai, toh pura tree traverse karna padega!

Solution: Data ko flat structure mein store karo - entities aur IDs separately. Jaise database mein tables hoti hain!

‚ùå Nested Structure (Slow)

{ posts: [ { id: 1, title: "Post 1", author: { id: 1, name: "John" }, comments: [ { id: 1, text: "Nice" } ] } ] }

‚ö†Ô∏è Update karna difficult!

‚úÖ Normalized (Fast)

{ posts: { ids: [1], entities: { 1: { id: 1, title: "Post 1", authorId: 1, commentIds: [1] } } }, users: { ids: [1], entities: { 1: { id: 1, name: "John" } } } }

‚úÖ Fast lookups & updates!

6Ô∏è‚É£ Immutable Updates - Immer.js Magic

Immer.js: A library that allows you to work with immutable state in a more convenient way by using "draft" state. Redux Toolkit includes Immer by default, allowing you to write "mutating" logic that actually produces immutable updates.

Purani problem: Redux mein state ko directly mutate nahi kar sakte. Har baar naya object/array banana padta tha using spread operator (...). Complex nested updates bahut messy ho jate the!

RTK solution: Immer.js built-in hai! Aap directly state ko modify kar sakte ho (mutable syntax), but behind the scenes Immer immutable updates kar raha hai! üé©‚ú®

‚ùå Manual Immutability (Complex)

// Nested update - Bahut complex! return { ...state, users: { ...state.users, [userId]: { ...state.users[userId], profile: { ...state.users[userId].profile, name: newName } } } }; // Array update return { ...state, todos: state.todos.map(todo => todo.id === id ? { ...todo, completed: true } : todo ) };

‚úÖ Immer (Simple & Clean)

// Same nested update - Super simple! state.users[userId].profile.name = newName; // Array update - Direct! const todo = state.todos.find(t => t.id === id); todo.completed = true; // Array methods work! state.todos.push(newTodo); state.todos.splice(index, 1); state.items.sort((a, b) => a - b);
// Immer in Action - All valid in RTK! const todosSlice = createSlice({ name: 'todos', initialState: { items: [], filter: 'all' }, reducers: { // ‚úÖ Direct mutation (Immer handles it!) todoAdded(state, action) { state.items.push(action.payload); }, // ‚úÖ Direct property update todoToggled(state, action) { const todo = state.items.find(t => t.id === action.payload); if (todo) { todo.completed = !todo.completed; } }, // ‚úÖ Array methods work! todoDeleted(state, action) { const index = state.items.findIndex(t => t.id === action.payload); state.items.splice(index, 1); }, // ‚úÖ Sort in place todosSorted(state) { state.items.sort((a, b) => a.text.localeCompare(b.text)); }, // ‚ö†Ô∏è Can also return new state (both work!) todosCleared(state) { return { items: [], filter: 'all' }; } } });

Q3. createAsyncThunk kya hai aur kab use karte hain? Lifecycle actions explain karo (Intermediate) ‚ñº

Answer:

createAsyncThunk: Async operations (API calls) handle karne ke liye utility. Ye automatically promise lifecycle actions generate karta hai.

Three Lifecycle Actions:

  1. pending: Jab async operation start hoti hai
  2. fulfilled: Jab successfully complete ho jaye
  3. rejected: Jab error aaye
// Create async thunk export const fetchUser = createAsyncThunk( 'users/fetchUser', async (userId, { rejectWithValue }) => { try { const response = await fetch(`/api/users/${userId}`); return await response.json(); } catch (error) { return rejectWithValue(error.message); } } ); // Handle in slice extraReducers: (builder) => { builder .addCase(fetchUser.pending, (state) => { state.loading = true; state.error = null; }) .addCase(fetchUser.fulfilled, (state, action) => { state.loading = false; state.user = action.payload; }) .addCase(fetchUser.rejected, (state, action) => { state.loading = false; state.error = action.payload; }); }

When to Use:

  • API calls (GET, POST, PUT, DELETE)
  • File uploads
  • Any async operation that needs Redux state

Hinglish: Jab bhi API call karni ho, createAsyncThunk use karo. Ye automatically loading, success, error states manage kar leta hai!

Why Normalize?

  • Performance: O(1) lookups by ID
  • No Duplication: Same data multiple places mein nahi
  • Easy Updates: Direct ID se update karo

Q6. Immer.js kaise kaam karta hai RTK mein? Kya limitations hain? (Advanced) ‚ñº

Answer:

Immer.js: Library jo "draft" state provide karti hai. Aap draft ko mutate kar sakte ho, Immer automatically immutable updates produce karta hai.

How it Works:

  1. Immer creates a "draft" copy of state (Proxy object)
  2. You mutate the draft directly
  3. Immer tracks all changes
  4. Produces new immutable state based on changes
// Without Immer (Old way) return { ...state, user: { ...state.user, profile: { ...state.user.profile, name: action.payload } } }; // With Immer (RTK way) state.user.profile.name = action.payload; // Simple! // Array operations state.todos.push(newTodo); // ‚úÖ Works! state.todos.splice(index, 1); // ‚úÖ Works! state.items.sort((a, b) => a - b); // ‚úÖ Works!

Important Rules:

  • ‚ùå Don't mix: Either mutate draft OR return new state, not both
  • ‚ùå Don't return undefined: Return nothing or new state
  • ‚úÖ Can return: Completely new state to replace
// ‚ùå Wrong - Mixing both state.value += 1; return { ...state }; // Don't do this! // ‚úÖ Correct - Mutate only state.value += 1; // ‚úÖ Correct - Return only return { value: state.value + 1 }; // ‚úÖ Correct - Replace entire state return initialState;

Limitations:

  • Only works in reducers (not in thunks or components)
  • Can't use with non-serializable data (functions, Promises)
  • Performance overhead for very large states

Hinglish: Immer magic ki tarah hai - aap mutable code likhte ho, but output immutable hota hai. Ye Proxy objects use karta hai changes track karne ke liye!

Best Practices:

  1. Feature Slices: Har feature ki apni slice, self-contained
  2. Colocate: Related code ek saath rakho
  3. Barrel Exports: index.ts files se clean imports
  4. Typed Hooks: useAppDispatch, useAppSelector banao
  5. Selectors: Reselect use karo memoization ke liye
  6. RTK Query: API logic separate services folder mein
// app/hooks.ts - Typed Hooks import { useDispatch, useSelector } from 'react-redux'; import type { RootState, AppDispatch } from './store'; export const useAppDispatch = () => useDispatch<AppDispatch>(); export const useAppSelector: TypedUseSelectorHook<RootState> = useSelector; // features/posts/index.ts - Barrel Export export { default as postsReducer } from './postsSlice'; export * from './postsSlice'; export * from './postsSelectors'; export * from './postsThunks';

Why Feature-Based?

  • Easy to find related code
  • Better for team collaboration
  • Easier to add/remove features
  • Scales well with app growth
  • Clear separation of concerns

Hinglish: Feature-based structure mein har feature ek mini-app ki tarah hai. Naya developer easily samajh sakta hai code kahan hai. Type-based structure (actions/, reducers/) mein sab scattered ho jata hai!
